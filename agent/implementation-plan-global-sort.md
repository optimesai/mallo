- **문제 요약**
  - 목록 정렬이 현재 페이지 내부 배열에서만 적용되고 전체 목록 기준으로 재정렬되지 않는다.
  - BOM 버전 배지(`v1.0`, `v1.4`, `v2.0` 등)가 전체 목록 기준으로 자연 정렬되지 않는다.
  - 수불 유형 배지(`생산불출`, `입고적재` 등)가 화면 표시 라벨 기준으로 정렬되지 않는다.

- **원인 진단**
  - 프론트엔드가 서버 500 오류 회피를 위해 정렬 파라미터를 고정값으로 전달하고 있다.
  - 일부 화면은 현재 페이지 응답 배열만 computed로 정렬한다.
  - BOM 그룹과 작업지시는 응답 DTO의 집계/계산 필드가 포함되어 JPA 엔티티 필드 정렬만으로 처리할 수 없다.
  - 수불 유형은 서버 enum 값과 화면 한글 라벨의 사전식 순서가 다르다.

- **구현 방침**
  - 프론트엔드는 클릭한 정렬 키와 방향을 `sort={field},{direction}` 형태로 서버에 전달한다.
  - 백엔드는 화면 DTO 정렬 키를 안전한 엔티티 경로나 DTO 정렬 로직으로 매핑한다.
  - BOM 그룹과 작업지시는 전체 조회 결과를 DTO 기준으로 정렬한 뒤 페이지 단위로 자른다.
  - 입고 목록은 DTO 필드를 JPA 엔티티 경로로 변환한 `Pageable`을 사용한다.
  - 수불 이력의 `transactionType` 정렬은 화면 한글 라벨 기준 순서로 정렬한다.

- **검증 계획**
  - 프론트엔드 `npm run build`로 타입 검사와 번들 빌드를 검증한다.
  - 백엔드 `./gradlew compileJava`로 Java 컴파일을 검증한다.
  - 정렬 요청 파라미터가 다시 동적 정렬값으로 전달되는지 코드 기준으로 확인한다.
